# Redstart — Changelog

Rolling, human-readable log of changes to the Redstart Nest / Redstart Twig codebase (formerly Beaver Dam / Beaver Log — see the 2026-07-13 rebrand entry). Not a replacement for `git log` — this is for context that helps troubleshooting later: *why* something changed, not just *what* diffed. Newest entries at the top.

---

## 2026-07-18

- **Fixed the File System capability IPC channel mismatch (the pre-existing Electron bug flagged in the chat-store refactor entry below).** The folder-scoped capability handlers are registered in a loop over `['vault', 'git', 'file_system']` that built the channel name straight from the key, so `file_system` registered `capabilities:set-file_system` / `capabilities:select-file_system-folder` (underscore) while the preload invokes the hyphenated `capabilities:set-file-system` / `...select-file-system-folder` — "No handler registered." Vault and Git were unaffected because their keys have no underscore. Fix: derive a hyphenated `slug = cap.replace(/_/g, '-')` for the two channel strings while keeping the underscore key for `setCapabilityConfig`. Two-line change, storage layer untouched. File System can now be enabled and its folder selected in the launcher; this also unblocks the MCP tool-calling path that couldn't be smoke-tested during the chat-store refactor.

- **Decomposed the Electron main-process god-file (`electron/main/index.mjs`) into per-namespace IPC modules — the main-process analog of the chat-store refactor below.** The single ~1,321-line file mixed app/window bootstrap, config/gateway builders, and 41 `ipcMain.handle` registrations across 10 namespaces. The 41 handlers now live in nine focused modules under `electron/main/ipc/`, the gateway-config builders moved to `gateway-config.mjs`, and `index.mjs` is **692 lines** (bootstrap, icon/CSP/window lifecycle, and a thin `registerIpcHandlers(deps)` dispatcher). Purely a maintainability refactor: no behavior, features, or IPC channel names changed.
  - **`ipc/<namespace>.mjs`** — one module per namespace, each exporting `register<Namespace>Handlers(deps)`: `github` (1), `hardware` (2), `settings` (4), `auth` (3), `profiles` (5), `tools` (7), `mcp` (4), `capabilities` (10 incl. the fixed folder loop), `server` (5). `tools:estimate-context` was reunited with the `tools` namespace (it had lived among the capabilities handlers).
  - **Dependencies flow one way via `deps`.** `setupIpcHandlers()` builds a single `deps` object and passes it to the registrars; modules import their own storage/gateway/secrets collaborators directly and receive only index.mjs-local functions (`readSettings`, `buildGatewayConfig`, `refreshLiveToolsConfig`, `resolveBinary`, `buildArgs`, …) through `deps`. No module reaches back for an index.mjs global. Namespaces with no index.mjs-local needs (`github`, `auth`, `mcp`) take empty deps.
  - **Shared mutable server state threaded explicitly.** The `server` namespace owns the live llama-server process, whose state is also read by lifecycle code that stays in `index.mjs` (`refreshLiveToolsConfig`, `startDiscoveryBeacon`, `before-quit`). The three former module globals (`serverProcess` / `serverEma` / `lastServerConfig`) were consolidated into one `serverState = { process, ema, lastConfig }` object passed by reference to both sides; `mainWindow` (reassigned after registration) is read through a `getMainWindow()` getter rather than captured by value.
  - **Config builder extracted too (seam 7).** `buildGatewayConfig` + `refreshLiveToolsConfig` moved from `index.mjs` into a new `gateway-config.mjs`, bringing `index.mjs` to **692 lines**. `buildGatewayConfig` is pure over its inputs (collaborators imported directly) and exported as-is; `refreshLiveToolsConfig` closes over the live `serverState`, so it is exported as a `createRefreshLiveToolsConfig(serverState)` factory that `index.mjs` binds once — the no-arg `refreshLiveToolsConfig` dep the capabilities/tools handlers receive is unchanged. Confirmed no overlap with `tools-gateway.mjs`, which *runs* the HTTP gateway and *consumes* this config while `gateway-config.mjs` *produces* it.
  - **Method, not a rewrite.** Done as **seven commits, one seam per commit** (leaf namespaces → profiles → tools+mcp → capabilities → server last as the riskiest → gateway-config builder), each `node --check`-clean and smoke-verified before the next — a pure move-and-delegate. Now-dead imports were removed from `index.mjs` as each namespace's sole consumers left (e.g. `ipcMain`, `dialog`, `spawn`, `crypto`, the start-side gateway/mcp/mdns/proxy fns, and the tool-registry/secrets fns once the config builder moved).

- **Verified:** `node --check` clean on `index.mjs` and all nine `ipc/*.mjs` after every commit; all 41 handlers preserved (verified by literal count) with no duplicate channel registrations; the File System fix confirmed by channel-name match against the preload. Smoke-tested in dev mode after seam 1 (pattern validation) and continuing through the remaining seams.

- **Decomposed the chat-ui god store (`chat.svelte.ts`) into a thin facade over six focused sub-stores.** The single ~1,928-line `ChatStore` class had accreted six unrelated responsibilities — UI dialog/edit state, per-conversation runtime state, message persistence, the send/stream pipeline, message edit/regenerate/delete operations, and a pile of pure helpers — which made the streaming engine (the part most likely to change) impossible to touch without wading through dialog state and DB persistence. It's now a **323-line delegating facade** backed by modules under `redstart-nest/src/chat-ui/src/lib/stores/chat/`. Purely a maintainability refactor: no behavior, features, or public API changed.
  - **`chat-options.ts`** (stateless) — pure option/timing helpers: `getApiOptions`, `getConversationModel`, `getContextTotal`, `parseTimingData`.
  - **`chat-ui-state.svelte.ts`** (`ChatUiState`) — error dialog, edit mode, the pending system-prompt edit id, the draft-message stash, and the queued pending-message map.
  - **`chat-runtime.svelte.ts`** (`ChatRuntimeState`) — per-conversation transient state: loading / reasoning / streaming maps, processing state, abort controllers, and inactive-conversation-state garbage collection.
  - **`chat-message-repo.ts`** (stateless) — message create/persist orchestration over `DatabaseService` + `conversationsStore`.
  - **`chat-send.svelte.ts`** (`ChatSendController`) — the send pipeline: `sendMessage`, the ~340-line `streamChatCompletion`, generation stop/save, and KV-cache pre-encoding.
  - **`chat-message-ops.svelte.ts`** (`ChatMessageOps`) — edit / regenerate / delete / continue operations, which reuse the send controller to (re)generate replies.
  - **Composition & dependency direction.** The facade owns one instance of each and wires dependencies one way (facade → ops → send → runtime/ui → repo/options); collaborators are passed in via constructor/param, never imported upward, so there are **no circular imports** (verified with `madge` after every step). Reactive `$state` stays declared in the class that owns it; the facade re-exposes it via forwarding getters — and setters where the old field was writable (e.g. a Storybook story assigns `chatStore.isLoading`) — so reactivity is preserved without copying values across the boundary.
  - **Public API unchanged.** The singleton `chatStore` and every module-level named export keep identical names, signatures, and reactive behavior; the **~15 files** that import `$lib/stores/chat.svelte` were not touched.
  - **Method, not a rewrite.** Done as **nine commits, one seam per commit**, each verified green before starting the next — a pure move-and-delegate. The handful of pre-existing `chat.svelte.ts` type errors were moved *verbatim* into the sub-stores that carry the offending code rather than fixed, to keep each commit a clean relocation with no smuggled-in behavior change.

- **Verified:** `npm run build` green after every commit; `svelte-check` error count held steady at its pre-refactor baseline the whole way (the pre-existing errors migrated with the code they belong to; none were added or removed); `madge --circular` clean on `src/lib/stores/chat` throughout. Smoke-tested end-to-end after each seam — send + stream + stop mid-stream, conversation switching mid-stream, per-chat loading indicators, context/processing readout, new-conversation-with-system-prompt, pending-message queueing, regenerate, continue, and all edit/delete variants. One path — MCP tool-calling — could not be exercised end-to-end because of an **unrelated pre-existing Electron bug** (the preload invokes `capabilities:set-file-system` / `capabilities:select-file-system-folder` but no `ipcMain.handle` is registered for them, so the File System tool can't be enabled in the Electron app); the agentic invocation itself moved verbatim and is functionally unchanged.

## 2026-07-17

- **Security hardening pass: auth is now on by default, localhost bypass removed, beacon payload minimized, Open Chat removed.** The session shifted the security posture from "opt-in auth with a localhost escape hatch" to "authenticate or don't get in," and stripped the beacon of every field that wasn't strictly needed for discovery.
  - **`accounts-storage.mjs` defaults `authRequired` to `true`.** New installations start with login enforced; legacy installs that have no `authRequired` field or have it unset are normalized to `true` on first read. The one-time owner-promotion migration (pre-existing `role: 'admin'` → `'owner'`) is preserved and still idempotent.
  - **Removed localhost bypass from `authenticate()`.** The old `isLocalhost()` short-circuit was deleted from `auth.mjs` — previously it let any request from `127.0.0.1` / `::1` skip token resolution entirely, which meant a logged-in admin at the physical console was always treated as anonymous and got 403 from admin-only routes. The fix from the 2026-07-09 session (token resolution now runs first) made the bypass unnecessary and actively harmful: with auth on by default, it was the only path to anonymous local access. Now every client, including the host machine's browser, must present a valid token or API key.
  - **Beacon payload minimized.** `beacon.mjs` (extracted from `index.mjs` into its own module) now returns only `{ running, port }`. The old payload leaked version, `authRequired`, the MCP server URL, and the LAN IP — none of which a discovery client needs. The minimal payload still supports the existing client-side `data.app === 'redstart-nest'` check in all three clients (Android `NetworkDiscoveryPlugin.java`, Windows `main.mjs`/`main.cjs`).
  - **Open Chat feature removed.** The captive Electron `BrowserWindow` and the local proxy server that served the chat UI through `redstart-chat://` were removed. The chat UI is now accessed directly in any browser at `http://<host>:19080` from any device on the network. This simplifies the launch flow, removes a second HTTP server, and makes the browser-based login gate work consistently (the captive window was never gated by auth).

- **Expanded MCP capabilities: File System, SQLite, Vault, Git, Scholar, and `web_search`.** Six new capability providers and one new web tool were added to the built-in MCP server, all following the same provider-driven pattern established by Postgres and Documents.
  - **File System** (`fs-tool.mjs`): eight tools — `fs_read_file`, `fs_write_file`, `fs_edit_file`, `fs_list_directory`, `fs_search_files`, `fs_get_file_info`, `fs_create_directory`, `fs_delete_file`. General-purpose read/write access to a user-chosen folder. Path containment is enforced by `resolveWithinRoot()` in the new `path-scope.mjs` module, which resolves symlinks on the deepest existing ancestor and compares against the real (symlink-resolved) root — a plain `startsWith()` check would miss symlink escapes. Binary extension writes are blocked. Files are capped at 50 MB and 7500 read characters. Downloads are served via a new `/files/download` gateway endpoint so model-created files are retrievable from the chat UI.
  - **SQLite** (`sqlite-tool.mjs`): read-only SQL query, table listing, and column inspection against `.db`/`.sqlite` files in an admin-configured folder. Same `READ ONLY` transaction enforcement model as Postgres.
  - **Vault** (`vault-tool.mjs`): read-only access to a folder of markdown notes. Search by full-text query, read individual files, browse tags — useful for Obsidian vaults or any markdown knowledge base.
  - **Git** (`git-tool.mjs`): read-only repository context from local git repos in an admin-configured folder. Exposes status, recent commits, and uncommitted diffs.
  - **Scholar** (`scholar-tool.mjs`): search open academic literature via OpenAlex, arXiv, and PubMed APIs. Abstracts, citations, and open-access PDFs are saved into the Documents output folder. An optional venue/category whitelist restricts results.
  - **`web_search`** (added to `web-fetch-tool.mjs`): first-party search using each source's own API (Wikipedia OpenSearch, arXiv API, PubMed E-utilities, MDN, Stack Exchange). No third-party search engine is involved. The model picks a source and a query, never a URL, so it cannot redirect a search elsewhere.

- **SSRF guard and redirect validation in `web_fetch`.** When the whitelist is toggled off, `web_fetch` now blocks private and loopback addresses explicitly — RFC1918 ranges, `localhost`, `.local`, link-local, and IPv6 loopback are all rejected so the model cannot probe the LAN, the gateway, or a router admin page. Redirects are followed manually with hop-by-hop validation: every `Location` header is checked against the same URL policy before the request is made, so a whitelisted page cannot silently bounce the fetch to a disallowed domain.

- **Shared path-containment utility (`path-scope.mjs`).** All file-based capabilities now import `resolveWithinRoot()` from a single module instead of rolling their own containment checks. The implementation resolves symlinks on the deepest existing ancestor of the candidate path, then compares against the real root — this catches `..` traversal, absolute paths, drive-qualified Windows paths, and symlink escapes in one place. Windows case-insensitivity is handled by normalizing case before comparison.

- **Centralized tool/capability definitions (`tools-definitions.mjs`).** Built-in tools, groups, and capabilities were moved from inline arrays in `index.mjs` into a dedicated module. The registry is imported by `index.mjs`, `App.tsx`, and the chat-ui, so there is exactly one source of truth for what ships built-in.

- **Chat-ui context compaction service.** A new `context-compaction.service.ts` handles summarizing long tool-call histories so the model's context window doesn't fill up with raw tool output. This is wired into the agentic loop in `ChatMessageAgenticContent.svelte` and `agentic.ts`.

- **Security test suite.** Added `tests/unit/security.test.ts` (Vitest) covering path containment, SSRF bypass attempts, binary extension blocking, and file-size limits. Also added `scripts/test-path-scope.mjs` for manual verification of symlink escape scenarios.

- **Verified:** `tsc --noEmit` clean on `redstart-nest`, `svelte-check` clean (same single pre-existing unrelated error), `node --check` clean on all touched `.mjs` files, all touched JSON configs parse. `test-path-scope.mjs` passes all symlink escape cases.

- **Server-side conversation storage with account scoping.** Conversations moved from client-side IndexedDB to a new `conversations.json` file in the Electron `userData` directory, served via `/conversations/*` HTTP routes in the gateway. Each conversation is scoped to the logged-in account ID when auth is on; when auth is off, a device-specific UUID (`redstart-device-id` in `localStorage`) scopes conversations per-device. The chat-ui's `DatabaseService` was rewritten to call the server API instead of Dexie/IndexedDB — all existing call sites in `conversationsStore` and `chatStore` work without changes.

- **30-day auto-cleanup.** `cleanupOldConversations()` runs on server startup and removes any conversation whose `lastModified` is older than 30 days. The cleanup is additive — it never touches active conversations.

- **mDNS advertiser with configurable hostname.** Added `bonjour-service` (pure JS, no native bindings) to broadcast the server as `Redstart Nest` on the local network via mDNS. The advertised hostname defaults to `redstart.local` and is configurable per profile in the React UI's Network section. The QR code, server address display, and mDNS broadcast all respect the configured hostname. The advertiser starts after the gateway is up and stops cleanly on server stop or app quit.
