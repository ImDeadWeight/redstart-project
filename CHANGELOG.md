- **Added a three-tier account role model: Owner / Admin / User.** Previously `role` was just `'admin' | 'user'` with a flat gate ‚Äî any admin-tier account could create, delete, or reset any other account, including other admins. Doesn't fit the target business use case (one sys-admin who bootstraps the install and controls Admin lifecycle; IT-staff sub-admins who manage regular Users but can't touch each other or the sys-admin). Explicitly scoped to this fixed hierarchy rather than a generic custom-permission system, which was discussed and deferred as unnecessary complexity for now.
  - `accounts-storage.mjs`: account records gain `status` (active/disabled), `createdBy`, `lastLoginAt`. One-time migration in `read()`: any pre-existing `role: 'admin'` account is promoted to `'owner'` the first time it's read after upgrade, as long as no owner exists yet ‚Äî preserves whatever access an install already had. `hasAnyAdmin()` renamed to `hasOwner()` since it now specifically gates the one-time Owner bootstrap.
  - `auth.mjs`: new `canManage(actor, targetRole)` gate ‚Äî Owner manages Admins and Users, Admin manages Users only, nobody manages the Owner (no transfer-ownership flow yet; recovery path is the same physical-access bootstrap in Beaver Dam's launcher). `listAccounts`, `createAccount`, `deleteAccount`, `resetPassword`, `regenerateApiKey` now take the actor's account and enforce actor-vs-target, not just actor-vs-flat-role ‚Äî an Admin can no longer create, delete, or reset another Admin by calling the route directly. `listAccounts(actor)` filters the result to User-tier accounts for Admin viewers (least visibility ‚Äî sub-admins don't see who the other admins are). `createFirstAdmin` renamed `createOwner`, kept structurally separate from `createAccount` (no "allow owner" escape hatch to misuse) and only reachable from Beaver Dam's own launcher IPC, never a route.

- **Fixed the chat-ui login gate never showing when "Require login" is on.** Most of the auth UI was already fully built in an earlier session (`LoginForm.svelte`, `authStore`, the `SettingsChatAccountsTab.svelte` admin UI, admin-only section gating) but had never been walked through end-to-end. Root cause: `+layout.svelte`'s `onMount` called `serverStore.fetch()` (hits `/props`) *before* `authStore.init()` (hits `/auth/config`) resolved, so any device without a session token sent a doomed unauthenticated request whenever login was required. The resulting 401 was thrown as a plain `Error` ‚Äî identical to a real connectivity failure ‚Äî and rendered by `ChatScreenServerError`, which only lives inside the app shell, i.e. only reachable when the login gate *didn't* trigger. Net effect: a 401 always looked like "server unavailable," never like "please log in."
  - `api-fetch.ts`: `apiFetch`/`apiFetchWithParams` now throw a new `ApiError` (carries the HTTP status) instead of a plain `Error`, so a 401 is distinguishable from a network failure.
  - `auth.svelte.ts`: added `authStore.handleUnauthorized()` ‚Äî clears the stale token/user so the reactive login gate reappears.
  - `server.svelte.ts`: on a 401 from `/props`, calls `handleUnauthorized()` instead of just recording a generic error string. Same mechanism covers a session dying mid-use (server restarted, admin revoked the account), not just first load.
  - `+layout.svelte`: `authStore.init()` now runs before `serverStore.fetch()`, and the props fetch is skipped entirely if login is required and no session exists yet ‚Äî the doomed request never fires. Added a `$effect` that re-fetches server props once `authStore.user` transitions to logged-in, since nothing previously retried that fetch after a successful login.

- **Added a Disable Mmap (`--no-mmap`) checkbox** next to Process Priority. llama-server's own log already warns `tensor overrides to CPU are used with mmap enabled - consider using --no-mmap for better performance` whenever `--fit` computes a CPU-offloaded MoE tensor placement ‚Äî i.e. exactly the scenario the auto-fit change above creates ‚Äî so this is a direct, binary-sourced recommendation tied to that mechanism rather than a speculative tuning knob. Off by default; purely additive (`noMmap?: boolean` on `LlamaConfig`, one conditional push in `buildArgs()`).
- **Stopped Beaver Dam from computing its own GPU-layer/MoE-offload split and let llama-server's built-in `--fit` handle it instead.** Found while investigating a CPU/GPU bottleneck reported during a training session: `buildArgs()` always passed explicit `-ngl` and `--n-cpu-moe` values, computed once in JS from total (not free) VRAM using a flat 256MB/layer guess ‚Äî and `--n-cpu-moe` was being given the CPU's physical core count as if it were a layer count, which is a unit mismatch (`--n-cpu-moe N` means "keep the first N MoE layers on CPU", confirmed via `llama-server.exe --help` and the vendored `common/fit.cpp`). Per `common/fit.h`, the binary's own `--fit` (on by default) only auto-adjusts arguments still at their default value ‚Äî so Beaver Dam's explicit values were silently defeating a strictly better mechanism that queries live free VRAM and the model's real tensor sizes at load time. `gpuLayers()`/`nCpuMoe()` heuristics deleted from the `profiles:generate-defaults` handler in `index.mjs`; `gpuLayers`/`nCpuMoe` are now left `undefined` by default (flag omitted ‚Üí auto-fit engages) with the existing UI fields still available as manual overrides. Also removed a same-class hardcoded `gpuLayers: 24` guess in `App.tsx`'s `scanHardware()`.
- **Disabled Electron hardware acceleration** (`app.disableHardwareAcceleration()` in `index.mjs`) ‚Äî the launcher/chat windows are plain UI with no WebGL/canvas-heavy rendering (Mermaid renders to SVG), so there's no reason for Chromium's GPU compositor to compete with llama-server for the same CUDA device.
